1. 这个项目到底是什么？

它不是 Codex 本身，也不是飞书本身。它是夹在两者中间的“小翻译 + 小调度员”。 飞书只懂聊天消息，Codex 只懂命令行任务；桥接项目负责把聊天消息变成命令行输入，再把命令行输出变成飞书回复。

你 在飞书里输入自然语言，比如“帮我解释这段代码”。
飞书机器人 接收你的消息，并通过飞书开放平台把事件送到本机服务。
本机桥接服务 一个 Node.js 程序，负责筛选消息、排队、调用 Codex、回传结果。
Codex CLI 真正执行任务的本地命令行工具，能读项目、跑命令、修改文件。
session 文件 记住每个 chat 对应的 Codex 对话，让下一句能接着上一句说。
screen 后台会话 让桥接服务一直开着，不然你关掉终端后机器人就不听消息了。

2. 用“群聊天”理解软件服务间的通话

对编程小白来说，最容易理解的模型是：每个软件服务都是群里的一个成员，只是它们不一定用中文聊天，有的用事件，有的用 JSON，有的用命令行。

你可以把整个系统想象成一个工作群：你是老板，飞书机器人是前台，桥接服务是项目经理，Codex 是工程师，session 文件是会议纪要。
老板：你 “请帮我修这个 bug。”
前台：飞书机器人 “收到一条用户消息，我转给项目经理。”
项目经理：bridge “这条消息有效，先排队，再找工程师处理。”
工程师：Codex “我读代码、跑命令、生成回答。”
会议纪要：session “上次聊到哪里了，我帮大家记住。”
回到群里 “工程师的最终结论发回飞书。”

5. 文件分工：小白应该先看哪几个文件

src/index.js
总指挥。连接飞书、监听消息、判断是否处理、排队、调用 Codex、回传答案。
src/codex.js
Codex 专员。负责启动 codex 命令、把 prompt 写进 stdin、读取 JSONL 输出。
src/image.js
图片助手。识别生图命令、构造 Codex 图片任务、校验 Codex 产出的本地图片。
src/sessions.js
记忆本。把每个 chat 的 Codex session id 存到 ~/.codex-feishu-bridge/sessions.json。
launch-wrapper.sh
启动脚本。设置 PATH 和环境变量，然后启动 Node 服务。
.env.example
配置模板。真正的 Secret 不应该写进项目文件。
SPEC.md
项目规格。说明目标、配置、运行行为、测试方式和安全默认值。

阅读顺序建议

  1. 先看 SPEC.md，知道项目要解决什么问题。
  2. 再看 src/index.js 的 handleMessage()，理解飞书消息进来后怎么处理。
  3. 再看 src/codex.js 的 runCodex()，理解怎么调用 Codex。
  4. 如果关心生图，再看 src/image.js，理解图片命令和图片路径校验。
  5. 最后看 src/sessions.js，理解为什么连续对话能接上。

6. 关键代码翻译成人话

飞书消息入口

channel.on({
  message: (msg) => {
    void handleMessage(msg)
  }
})

人话：飞书每送来一条消息，就调用 handleMessage。这里的 msg 就是“快递包裹”，里面有 chat id、发送人、消息内容、是否 @ 机器人等信息。

判断要不要处理

if (msg.chatType !== "p2p" && config.requireMentionInGroup && !msg.mentionedBot) return;

人话：如果不是私聊，而且配置要求群里必须 @，但这条消息没 @ 机器人，那就不理。这样机器人不会偷听整个群的所有聊天。

调用 Codex

const result = await runCodex({
  prompt: text,
  sessionId: previousSessionId,
  workdir: config.workdir,
  sandbox: config.sandbox
});

人话：把你在飞书里说的话交给 Codex。如果之前这个 chat 已经聊过，就带上 session id，让 Codex 继续同一个对话。

回传飞书

await channel.send(msg.chatId, { markdown: result.text }, { replyTo: msg.messageId });

人话：把 Codex 最终回答发回同一个飞书 chat，并且回复到你的原消息下面。

图片回传

await sendImageReply(msg, imageResult.path);

人话：如果你发的是 /image、生图、画一张 这类命令，桥接会要求 Codex 把图片保存成本地文件，再上传成飞书图片消息。

7. 为什么需要 session？

如果没有 session，你每发一句，Codex 都像第一次见你。session 就像“会议纪要”，让 Codex 记得上轮说过什么。

没有 session 你说“继续刚才那个方案”，Codex 可能不知道“刚才”是什么。
有 session 你说“继续”，桥接服务把旧 session id 带给 Codex，Codex 能接上上下文。 
{
  "oc_xxx_chat_id": {
    "sessionId": "019e635a-...",
    "updatedAt": 1779783591518
  }
}

8. 私聊、群聊、命令分别怎么工作？

私聊 直接发消息 任何文本都会进入 Codex，适合个人使用。
群聊 默认必须 @ 群里只有 @ 机器人时才处理，避免误触发。
命令 /new 和 /status /new 清空当前 chat 的记忆；/status 查看运行状态。
生图 /image、/img、画一张 生图命令会让 Codex 产出本地图片，再由桥接上传回飞书。

9. 你怎么知道是 Codex 在回答？

当前项目里，普通文本消息只会调用 runCodex()，所以“Codex 执行中。”后面的最终回复就是 Codex CLI 的回答。 另外，最终回复会带 Codex · session ... 页脚，发 /status 也可以看到当前使用的 codex-cli 版本。

图片消息本身不能直接带长文本，所以桥接会先发图片，再补一条带 Codex · session ... 的说明。

10. 现在这个版本的边界

  • 支持文本消息和 Codex 产出的本地图片回传；仍不处理用户上传的图片、文件、语音。
  • 只回传最终答案，不实时显示 Codex 工具调用过程。
  • 每个 chat 串行执行，防止同一个项目同时被多个任务改乱。
  • 默认 sandbox 是 workspace-write，Codex 可以写工作区，但不会默认全盘无沙箱。
  • 默认不需要 OPENAI_API_KEY；如果未来让桥接服务直接调用 OpenAI Images API，才需要单独配置。
  • 如果 Codex 里再调用 lark-cli，可能遇到 keychain 或 sandbox 权限问题，这是另一个层面的配置。
